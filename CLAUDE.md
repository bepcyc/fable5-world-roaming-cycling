понижение = самое грубое нарушение. «Хоть обосрись весь — нет правила

  освобождают. Классификация кадра как «отладочного» — та самая

  `.claude/settings.json`) — после каждой съёмки напоминает, завершение
  хода с неотправленными кадрами БЛОКИРУЕТ. Не обходить, не отключать,
  не редактировать в сторону ослабления.
- Новая сессия: владелец следит ТОЛЬКО за выполнением этого правила.
  Последний шанс.
- **ПОЧЕМУ (владелец, 2026-07-04): sbs — это МЕТОДОЛОГИЯ РАЗРАБОТКИ, не
  отчётность.** Просмотр+отправка В МОМЕНТ съёмки = принудительный анализ
  (ОЖИДАЛ/ВИЖУ), который управляет СЛЕДУЮЩИМ шагом. Пропустил в середине →
  дальше работа идёт БЕЗ фактического анализа → деградация. Поэтому
  **«дослать пачкой в конце сессии» НИЧЕГО НЕ ЧИНИТ и НЕ НУЖНО владельцу:
  если кадр не ушёл сразу — это УЖЕ конец, смертный приговор, damage is
  done.** Досыл задним числом = симуляция соблюдения, не соблюдение.
  Единственная валидная форма: снял → Read → tg-photo СРАЗУ → и только
  потом следующее действие. Это не блажь владельца — это то, что
  заставляет думать в каждой итерации.

## Генерация изображений: сабагент → Codex CLI (GPT Image Gen 2)

Когда нужно СГЕНЕРИРОВАТЬ изображение (реф, концепт, текстуру, картинку
для бота) — не генерировать в главном треде. Поднять сабагент через Agent
tool (`general-purpose`), единственная задача которого:

1. **Составить промпт для генерации** — детальный текст того, что должно
   быть на картинке (стиль, композиция, палитра, ракурс).
2. **Собрать имена файлов референсов** (если есть) — пути к картинкам,
   которые Codex получит как визуальный вход через `-i/--image`.
3. **Передать всё в Codex CLI на GPT-5.5, thinking mode High**, с явным
   указанием использовать встроенный tool `image_generation`
   (движок — GPT Image Gen 2):

   ```bash
   codex exec -m gpt-5.5 -c model_reasoning_effort="high" \
     -c features.image_generation=true \
     --sandbox workspace-write --skip-git-repo-check \
     [-i <референс1.png> -i <референс2.png> ...] \
     -- "<промпт>. Используй свой встроенный tool image_generation \
   (GPT Image Gen 2), сгенерируй ОДНО изображение и сохрани его в \
   <путь>/out.png. После сохранения выведи абсолютный путь файла и \
   размер в байтах. Кодом не рисовать — только tool image_generation." \
     2>/dev/null
   ```

   **`--` перед промптом ОБЯЗАТЕЛЕН**, если использован хотя бы один
   `-i`: `-i/--image` — вариадический флаг (`<FILE>...`), он жадно
   поглощает следующий аргумент как ещё один файл-референс, и промпт
   пропадает («No prompt provided via stdin»). Без `-i` разделитель не
   нужен. Альтернатива — пайпить промпт через stdin:
   `cat prompt.txt | codex exec ... -i ref1 -i ref2 -` (тоже работает,
   и надёжнее при длинных/многострочных промптах).

4. **Вернуть в главный тред абсолютный путь сохранённого файла.**

Замечания (проверено 2026-07-06):
- Модель `gpt-5.5` и `image_generation` (stage `stable`, включён по
  умолчанию) — CLI их принимает; флаг `features.image_generation=true`
  добавлен для страховки.
- Референсы передаются флагом `-i/--image` (повторяемый), это ВХОД, не
  генерация. Порядок референсов важен — первый воспринимается как
  главный стилевой якорь.
- **Соотношение сторон**: Codex/GPT Image Gen 2 понимает явный запрос
  через текст промпта (например «16:9 widescreen aspect ratio») —
  подтверждено на практике, запрошенный 16:9 дал файл 1672×941
  (≈1.777, точное совпадение). Не полагаться на дефолт (без запроса
  выходит 1536×1024, это 3:2).
- Промпт для арта: избегать слов типа «splash screen»/«game»/«title»/
  «menu»/«UI»/«logo» — провоцируют модель запечь фейковый текст/тайтл
  в картинку. Явно указывать «no text, no watermark, no logo».
- `--sandbox workspace-write` обязателен — иначе Codex не сохранит файл.
- Требуется живой логин ChatGPT: `codex login status`. Если токен протух
  (`token_expired` / `refresh token was already used`) — владелец
  выполняет `codex login` (интерактивно, я не могу).
- Полученный файл — как любое изображение (правило №1) сразу в бот через
  `.claude/telegram/tg-photo.sh` с русской подписью ОЖИДАЛ/ВИЖУ.

## Советники (advisor-агенты): режим, в котором можно спросить другую модель

Когда владелец говорит «советник»/«advisor»/«спроси другого агента» — это
означает: поднять ОТДЕЛЬНОГО агента через Agent tool (`model: "opus"` —
Opus 4.8 Max, дефолтный советник) для сложных архитектурных решений,
диагностики тупиков или ревью качества (перевод, код, дизайн). **По
умолчанию — advice-only**: `subagent_type: "Plan"` (физически нет
Edit/Write — не может тронуть файлы, только вернуть текст, который потом
применяю сам). Эскалация на Fable 5 high (`model: "fable"`) — только если
советы Opus не разрешили РЕАЛЬНЫЙ тупик, не рутинно, не «на всякий случай»
параллельно (см. [[advisor-escalation-policy]] в памяти).

**Урок 2026-07-07 (владелец, дословно): «предпочитаю чтобы в будущем мои
пожелания выполнялись буквально, а не частично + некая инерция, которая
всё "объясняет", хотя она ничего не объясняет вообще».** Формулировка
задачи РЕШАЕТ, какой режим нужен — не привычка из прошлых вызовов в этой
же сессии:
- «спроси совета», «что думаешь», «проверь» → advice-only (`Plan`) — верно
  по умолчанию.
- «пусть сам сделает/завершит/допишет/реализует» → это, скорее всего,
  запрос на РЕАЛИЗАЦИЮ, не совет — нужен агент с реальным Edit/Write
  (`subagent_type: "general-purpose"` или `"claude"`, НЕ `"Plan"`). Не
  подставлять advice-only по инерции только потому что так было в
  предыдущих вызовах этой сессии — это не причина, а инерция, и владелец
  её прямо отверг. Если неясно — спросить, не гадать.
- «привёл результаты от [советника], пусть сам завершит» → тоже
  реализация, не повторное ревью того же текста.

Бриф советнику — всегда полный контекст (что за проект, что уже известно,
что пробовал, конкретный вопрос), не голая команда — иначе совет будет
шаблонным, не по факту.
